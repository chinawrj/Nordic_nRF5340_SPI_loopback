### 3. First bsim compile ⚠️ → small code change → ✅
`west build -b nrf5340bsim/nrf5340/cpuapp --build-dir build-bsim`:

- **CMake config PASS** (once `BSIM_OUT_PATH` + `BSIM_COMPONENTS_PATH` are set, FindBabbleSim passes)
- **Compile FAIL**: `src/spi_loopback.c` has `DT_NODELABEL(spi4)` which does not exist on the bsim board DTS (bsim has no SPIM4 peripheral)

**Fix**: in `spi_loopback.c` wrap all SPIM4-related code in `#if DT_NODE_EXISTS(SPI_NODE) && DT_NODE_HAS_STATUS(SPI_NODE, okay)`; the `#else` branch turns `spi_loopback_start()` into a LOG_WRN + return 0 stub. So:
- On the physical DK board: SPIM4 loopback runs normally (AC-1 path unchanged)
- On the bsim virtual board: the SPI module is silently disabled, but the BLE half still compiles and runs (that's the meaning of AC-R2 — verifying the BLE-host code does not depend on hardware SPI)

Recompiled: **186/186 PASS**, produced `build-bsim/Nordic_nRF5340_SPI_loopback/zephyr/zephyr.exe` (native-simulator executable).

### Key learnings
1. **Manual BabbleSim clone is 100× faster than `west update`**: west does not support selective update of imported projects; once `+babblesim` enables the group filter, every project (including 800 MB connectedhomeip) is re-checked. Directly `git init + fetch --depth=1 + checkout <sha>` is strictly equivalent.
2. **Cross-board compatibility in embedded code**: `DT_NODELABEL` expands at compile time and errors hard if the label does not exist. Guarding at preprocessor level with `DT_NODE_EXISTS` + `DT_NODE_HAS_STATUS` lets the same source compile on both hardware and simulation boards. This is the Zephyr-community standard pattern (lots of examples in `zephyr/samples/...`).
3. **ext_libCryptov1 is bsim's legacy baggage**: openssl 1.0.2g is the 2016 version, gcc-13 can never build it; but it is an independent component in bsim and BLE simulation does not need it. When hit, just ignore.
