### 3. 首次 bsim 编译 ⚠️ → 代码小改 → ✅
`west build -b nrf5340bsim/nrf5340/cpuapp --build-dir build-bsim`：

- **CMake 配置阶段 PASS**（设 `BSIM_OUT_PATH` + `BSIM_COMPONENTS_PATH` 后 FindBabbleSim 通过）
- **编译阶段 FAIL**：`src/spi_loopback.c` 里 `DT_NODELABEL(spi4)` 在 bsim board DTS 上不存在（因为 bsim 没有 SPIM4 外设）

**修复**：在 `spi_loopback.c` 用 `#if DT_NODE_EXISTS(SPI_NODE) && DT_NODE_HAS_STATUS(SPI_NODE, okay)` 把全部 SPIM4 相关代码包起来；`#else` 分支里 `spi_loopback_start()` 变成 LOG_WRN + return 0 的 stub。这样：
- 物理 DK 板上：SPIM4 loopback 正常运行（AC-1 路径不变）
- bsim 虚拟板：SPI 模块静默禁用，BLE 部分仍然编译+可跑（这就是 AC-R2 的意义——验证 BLE host 代码不依赖硬件 SPI）

重新编译：**186/186 PASS**，产出 `build-bsim/Nordic_nRF5340_SPI_loopback/zephyr/zephyr.exe`（native simulator executable）。

### 关键学习
1. **BabbleSim 手动 clone 比 `west update` 快 100 倍**：west 不支持选择性更新 import 进来的项目；一旦 `+babblesim` 加上 group filter，全部项目（包括 800 MB 的 connectedhomeip）都会被重检。直接 `git init + fetch --depth=1 + checkout <sha>` 精确等价。
2. **嵌入式代码的跨 board 兼容**：DT_NODELABEL 在编译期展开，如果标签不存在会硬报错；用 `DT_NODE_EXISTS` + `DT_NODE_HAS_STATUS` 在预处理阶段守卫，才能让同一份源码在硬件板和模拟板都编译通过。这是 Zephyr 社区的标准套路（`zephyr/samples/...` 里大量例子都这么写）。
3. **ext_libCryptov1 是 bsim 的遗产包袱**：openssl 1.0.2g 是 2016 年的版本，在 gcc-13 上无论如何都编不过；但这是 bsim 里独立组件，BLE 模拟不依赖。碰到就忽略。
