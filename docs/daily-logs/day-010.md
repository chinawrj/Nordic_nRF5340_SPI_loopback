## Context

Day 9 locked the invitation gate: DK hardware clean build + bsim HRS runtime (AC-V1) + Chrome Web Bluetooth (AC-V2). Gate-HW was already green. Today's target: close **Gate-SIM-BSIM = AC-V1** — two bsim `zephyr.exe` processes communicating over BabbleSim phy, central receiving HRS notifications from our DUT.

Prior Day-10 attempts had stalled for several rounds on:
```
undefined reference to __device_dts_ord_41   // bt_hci_sdc
```
NCS v2.9.0 pins `BT_LL_CHOICE` default to `BT_LL_SOFTDEVICE`, and the SDC controller fails to link for any `*_bsim` target.

## Diagnosis

1. Confirmed upstream Zephyr `peripheral_hr` **also** fails to link on `nrf5340bsim` / `nrf52_bsim` inside the NCS v2.9.0 workspace — not project-specific, the SDC default breaks every bsim BLE sample.
2. Read `nrf5*.dtsi` directly and found two pre-declared HCI nodes on every SoC:
   - `bt_hci_sdc` (status=okay, SoftDevice Controller — the broken one)
   - `bt_hci_controller` (status=disabled, `compatible = "zephyr,bt-hci-ll-sw-split"` — Zephyr's open-source LL, works fine on bsim)
3. Solution: a DTS overlay that flips which HCI node is chosen, plus `CONFIG_BT_LL_SW_SPLIT=y`.

## Solution

### 1. Board pivot: `nrf5340bsim` → `nrf52_bsim`

Ran our existing AC-R2 artefact (`build-bsim/` from Day 7, nrf5340bsim DUT) against Zephyr's central test. DUT failed at runtime:
```
[00:00:01.000,030] <err> bt_hci_driver: Endpoint binding failed with -11
```
The bsim IPC back-end between App Core and Net Core is broken in NCS v2.9.0. Pivoted the DUT to single-core `nrf52_bsim`, which runs App + LL in one process.

**Trade-off acknowledged**: nrf52_bsim does not exercise the App↔Net IPC. That's OK — §6.1's hardware target proves dual-core (`build/ipc_radio/` + `merged.hex`); bsim proves BLE protocol behavior over the 2.4 GHz virtual phy. Two pieces of evidence combine to the full chain.

### 2. DTS overlay + Kconfig fragment

Committed to `scripts/bsim-hrs/`:

- `swll.overlay`:
  ```dts
  &bt_hci_sdc        { status = "disabled"; };
  &bt_hci_controller { status = "okay"; };
  / { chosen { zephyr,bt-hci = &bt_hci_controller; }; };
  ```
- `swll.conf`: `CONFIG_BT_LL_SW_SPLIT=y`

Applied identically to both DUT and central binaries.

### 3. Build + run scripts (checked in)

- `scripts/bsim-hrs/build.sh` — builds our DUT to `build-bsim52/` via `west build -b nrf52_bsim` with the overlay + Kconfig fragment, and builds Zephyr's `tests/bsim/bluetooth/samples/central_hr_peripheral_hr` via Zephyr's own `_compile` helper. Idempotent.
- `scripts/bsim-hrs/run.sh` — kills stale bsim processes, starts `bs_2G4_phy_v1 -D=2 -sim_length=15e6`, DUT as `-d=0`, central as `-d=1 -testid=central_hr_peripheral_hr`; writes logs to `reports/bsim-hrs-{phy,peripheral,central}.log`; greps `NOTIFICATION.*length 2`; fails if <3.

### 4. Run-from-bin-dir fix for phy

First pass: phy logged
```
p_2G4: ERROR: ../lib/lib_2G4Channel_NtNcable.so: cannot open shared object file
```
`bs_2G4_phy_v1` `dlopen`s channel/modem `.so` via a path that is relative to CWD, not to the binary. Fixed by `( cd "$BSIM_OUT_PATH/bin" && ./bs_2G4_phy_v1 ... )`.

## Lessons

- When NCS defaults force a DT node that won't link for your target, don't fight Kconfig — find the sibling DT node that is pre-declared but disabled, and flip it via overlay. Nordic ships both `bt_hci_sdc` and `bt_hci_controller` in every nrf5*.dtsi for exactly this reason.
- Always source-check upstream samples against the same workspace before blaming project config. If upstream `peripheral_hr` also fails to link, the bug isn't in your app.
- bsim's phy `dlopen`s `../lib/*.so` relative to **CWD**, not to argv[0]. Running from anywhere other than `$BSIM_OUT_PATH/bin` requires an explicit cd.
