### 4. GitHub Actions CI 查看 ⚠️ → 修复 ✅
**两次运行均 FAILED**（runs `24654121778` + `24655705928`）。

查日志定位根因：
```
2026-04-20T08:12:12.354Z /home/runner/.local/bin/nrfutil: line 5: syntax error near unexpected token ']'
2026-04-20T08:12:12.354Z /home/runner/.local/bin/nrfutil: line 5: `  } ]'
```
**与 Day 5 本地踩过的完全同一坑**：workflow 里用了老 URL `linux-amd64/nrfutil` → artifactory 返回 404 JSON body，`curl -sL` 不检测 HTTP 错误，把 JSON 下载成 "nrfutil"，bash 尝试当 script 运行立刻炸。

**修复 `.github/workflows/build.yml`** 四处：
1. **Install step 使用新 URL** — `x86_64-unknown-linux-gnu/nrfutil`；curl 加 `-f` 让 HTTP 错误退出非零
2. **Cache path 更正** — 从 `/opt/nordic/ncs/toolchains`（macOS 习惯）改为 `~/ncs/toolchains` + `~/.nrfutil`（Linux 实际路径，Day 5 实测）；cache key 加 `-v2` 让旧 cache 失效
3. **Env propagation 补齐** — 在 `GITHUB_ENV` 导出列表加入 `LD_LIBRARY_PATH` / `PYTHONPATH` / `PYTHONHOME`（NCS python 3.12 + 绑定 libcurl 运行时必须）
4. **apt prereqs 精简** — 去掉 `gcc-multilib/g++-multilib/libmagic1`（Day 5 实测在 noble security 镜像返回 404；ARM target 用不上 multilib；libmagic1 已更名 libmagic1t64 会作为传递依赖拉入）

YAML 语法校验 `python3 -c "import yaml; yaml.safe_load(...)"` → OK。

### 关键学习
1. **CI 是 Day 5 踩坑经验的第二次验证机会**：workflow 是 Day 4 在 macOS 上凭记忆写的，Day 5 用户改到 Linux 后，本地踩坑同时也暴露了 workflow 里同构的 bug。修 workflow = 把本地经验固化成自动化回归。
2. **`curl -sL` 的致命缺陷**：`-s` silent + `-L` follow redirects，但**不** fail on HTTP error。未来所有 CI 脚本默认用 `curl -fsSL`，让 HTTP 4xx/5xx 立刻退出。
3. **Cache key 策略**：把平台差异的路径写进 key 后缀（如 `-v2`）可以让旧 cache 平滑失效，不需要手动清理 GitHub Actions cache。
