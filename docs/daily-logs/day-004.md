### 任务 1: GitHub Actions Workflow
文件：`.github/workflows/build.yml`

设计：
- Ubuntu 22.04 runner
- **两级缓存**：
  - `ncs-toolchain-v2.9.0-Linux` — 缓存 `/opt/nordic/ncs/toolchains/` (1.5 GB)
  - `ncs-modules-v2.9.0-<hash(west.yml)>` — 缓存 `nrf/ zephyr/ nrfxlib/ modules/ bootloader/ tools/ test/ .west/` (数百 MB)
  - 命中时跳过 `nrfutil toolchain-manager install` 和 `west update`，预期从 15+ min 降到 ~3 min
- **NCS env 传递**：把 `nrfutil ... --as-script` 输出到文件再 source，提取 `PATH/ZEPHYR_*/GNUARMEMB` 等写入 `$GITHUB_ENV`，后续 step 免 source
- **双构建策略**：先跑常规 sysbuild（确保正常构建不回归），再跑 `-Werror` 严格构建（AC-Q1），最后 `verify-acceptance.sh` 断言
- **Artifact upload**：`merged.hex` 作为 CI artifact，方便面试官按 commit 下载验证

风险：
- 无法本地验证 GitHub-hosted Ubuntu runner 上 `nrfutil linux-amd64` binary 是否可以下载（需要 push 后才知道）。Day 5 第一件事：看 Actions tab 绿不绿，红了按日志修。
- `west update --narrow -o=--depth=1` 是为了加快初次 clone —— 这个组合在 NCS 上是官方文档推荐的。

### 技术笔记
- **CI 缓存 key 设计**：modules 用 `hashFiles('west.yml')` 让 west.yml 任何变更自动失效；toolchain 用固定 key（NCS 版本变了才失效），因为 toolchain 下载最慢。
- **sysbuild 与 Twister**：`sample.yaml` 里 `sysbuild: true` 让 Twister 走和 `--sysbuild` 相同的路径（否则 net core 不会被构建）。
- **--depth=1 风险**：shallow clone 偶尔会因 NCS 依赖的某个 manifest tag 在深处而失败；如遇到 `west update` CI 失败，第一反应是去掉 `--narrow -o=--depth=1` 换全量 clone。
- **交接文档的价值**：不是给面试官看的，是给"明天的自己在陌生机器上"看的。4 天的隐式上下文（为什么 west.yml 里要写 `path: nrf`、为什么 `CONFIG_BT_HCI_IPC` 不能写、zsh 为什么对 `eval "$(...)"` 有问题）全部显式化，省却第二次踩坑。
